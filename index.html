EXPLAINER — 01

WHAT IS A PROXY VARIABLE?

The sneaky reason AI stays biased even after you remove race from the data.

One-Sentence Definition

A proxy variable is a data feature that correlates with a protected attribute — like race, gender, or class — even though it doesn't mention it directly. So removing the protected attribute doesn't remove the bias it encodes.

Why This Matters

Most people assume that removing race from an AI model makes it race-neutral. This is wrong. Machine learning models don't care what a feature is called. They care what it predicts. If a feature like custody_status or zip_code is correlated with race in the training data, the model will learn and use that correlation — whether or not the word "race" appears anywhere in the dataset. This is called proxy discrimination, and it's one of the hardest problems in algorithmic fairness.

Common Proxy Variables
Feature Encodes Why
zip_code Race Historical redlining segregated neighbourhoods. Zip codes still reflect this.
custody_status Race Over-policing of Black communities leads to disproportionate pretrial detention.
name Gender / Ethnicity Names encode gender and ethnicity with high accuracy.
college_attended Socioeconomic class Elite colleges remain heavily stratified by class and race.
prior_arrests Race Arrest records reflect over-policing, not actual crime rates.
employment_tenure Age A 24-year-old structurally cannot have 10 years of work history.

How to Detect Proxy Variables
detect_proxy.py
from scipy.stats import chi2_contingency

def check_proxy(df, feature, protected_attr):
  """Returns p-value. If < 0.05, likely a proxy."""
  contingency = pd.crosstab(df[feature], df[protected_attr])
  chi2, p, dof, _ = chi2_contingency(contingency)
  return { 'p_value': round(p, 4), 'is_proxy': p < 0.05 }

# Run on every feature before training
result = check_proxy(df, 'CustodyStatus', 'race')
# → { 'p_value': 0.0001, 'is_proxy': True }
The Bigger Picture

Proxy variables exist because our social world is stratified. Zip codes encode race because of redlining. Custody records encode race because of over-policing. Employment tenure encodes age because time is linear. Data doesn't exist in a vacuum — it reflects the society that generated it. A model trained on that data will learn those reflections, unless you actively audit and intervene.

Algorithmic auditing is not optional. It is a prerequisite for deployment.

EXPLAINER — 02

WHAT IS SAMPLING BIAS?

The reason your AI works great in the lab and fails on the people who need it most.

One-Sentence Definition

Sampling bias occurs when the data used to train an AI system doesn't accurately represent the population it will be deployed on — so the model learns patterns that don't generalise, performing well for over-represented groups and failing for under-represented ones.

Why This Matters

Most people assume that if you have a large dataset, the model will be fair. This is wrong. Size doesn't equal representativeness. A dataset of 10 million records is still biased if 9.8 million of them come from the same demographic. The model doesn't know what it hasn't seen — it optimises for the distribution it was trained on. When that distribution doesn't match the real world, the people left out pay the price. This is how facial recognition fails on darker skin tones, clinical tools miss diagnoses in women, and NLP models perform worse on dialects outside Standard American English.

Common Forms of Sampling Bias
Form What It Means Real Example
Underrepresentation A group appears far less in training data than in the real world Facial recognition trained mostly on light-skinned faces
Selection Bias Data collected from a non-random subset of the population Clinical AI trained on records from affluent ZIP codes only
Historical Bias Past data reflects past inequities; the model learns the inequity Hiring models trained on 10 years of male-dominated engineering hires
Survivorship Bias Only "successful" cases make it into the dataset; failures are invisible Loan models trained only on approved applicants, never denied ones
Measurement Bias The same reality is measured differently for different groups Over-policing creates more arrest records for Black communities — not higher crime
Real-World Proof: MIT Gender Shades (2018)

Joy Buolamwini & Timnit Gebru tested three commercial facial recognition systems on a dataset balanced across gender and skin tone. Every system performed dramatically worse on darker-skinned women — not because the algorithms were programmed to fail them, but because training datasets were overwhelmingly composed of lighter-skinned faces. The fix was not a new algorithm. It was a more representative dataset.

Group Best System Error Rate Worst System Error Rate
Lighter-skinned males ~0.8% ~3.4%
Lighter-skinned females ~1.7% ~7.1%
Darker-skinned males ~10.8% ~16.0%
Darker-skinned females ~20.8% ~34.7%
A 34-percentage-point gap — from the same model, on the same task, because of who was and wasn't in the training data.

How to Detect Sampling Bias
audit_representation.py
def audit_representation(df, group_col, expected_proportions):
  """Flag groups as OVER/UNDERREPRESENTED vs real-world."""
  proportions = df[group_col].value_counts(normalize=True)
  audit = pd.DataFrame({'dataset_proportion': proportions})
  audit['expected'] = pd.Series(expected_proportions)
  audit['ratio'] = (audit['dataset_proportion'] / audit['expected']).round(4)
  audit['flag'] = audit['ratio'].apply(
    lambda r: 'OVERREPRESENTED' if r > 1.5
              else 'UNDERREPRESENTED' if r < 0.67
              else 'OK'
  )
  return audit

# Run before every training job
real_world = {'men': 0.51, 'women': 0.49}
result = audit_representation(df, 'gender', real_world)
# UNDERREPRESENTED → oversample, reweight, or restrict deployment
The Bigger Picture

Sampling bias is not a technical accident. It reflects who had access to the systems that generated the data in the first place. Facial recognition datasets skew lighter-skinned because the researchers who built them moved in predominantly lighter-skinned professional environments. Clinical AI skews wealthy because hospitals in affluent ZIP codes have better electronic health record infrastructure. NLP models skew toward Standard American English because that's what most of the training corpus is written in. The data you have is a record of the world as it was — including which people were visible, documented, and deemed worth measuring. A model trained on that data will reproduce those decisions at scale.

You cannot fix a representativeness problem with a better algorithm. You fix it with better data — or you disclose the limitation and restrict deployment accordingly.

WHAT ARE SHAP VALUES?

The reason "the model said so" is not an explanation — and what to do instead.

One-Sentence Definition

SHAP values (SHapley Additive exPlanations) tell you exactly how much each feature contributed to a specific model prediction — in a unit you can reason about — so you can see what actually drove a decision, not just what the decision was.

Why This Matters

Most AI systems deployed in high-stakes domains — criminal justice, hiring, lending, healthcare — are black boxes. They produce a score or a decision and don't explain why. This is not just an inconvenience. It is a fairness problem. If you cannot inspect how a model arrived at a decision, you cannot detect whether a protected attribute drove the outcome, challenge a decision that affected you, or comply with regulations that require explainability. SHAP values make the black box transparent. They are grounded in cooperative game theory (Shapley values), providing a mathematically principled way to attribute credit or blame across features — consistent, locally accurate, and model-agnostic.

SHAP vs Other Explainability Methods
Method Scope Consistent?
SHAP Local + Global ✓ Mathematically guaranteed — values sum exactly to the prediction
LIME Local only ✗ Approximate — results can vary between runs
Feature Importance Global only ✗ Biased toward high-cardinality features; no per-decision insight
Permutation Importance Global only Reasonable, but tells you nothing about individual decisions
Real-World Proof: COMPAS Analysis

We applied SHAP to the biased and fair COMPAS models to inspect what they were actually using — feature by feature, decision by decision.

How to Run a SHAP Bias Audit
shap_bias_audit.py
import shap
import pandas as pd

# 1. Fit explainer (fast for tree models)
explainer = shap.TreeExplainer(model)
shap_values = explainer.shap_values(X_test)

def shap_bias_audit(shap_vals, feature_names, protected):
  """Flag protected attributes / proxies in top drivers."""
  mean_abs = pd.Series(shap_vals.mean(axis=0), index=feature_names).abs()
  mean_abs = mean_abs.sort_values(ascending=False).reset_index()
  mean_abs.columns = ['feature', 'mean_abs_shap']
  mean_abs['flag'] = mean_abs['feature'].apply(
    lambda f: 'BIAS RISK' if any(p in f for p in protected) else '')
  return mean_abs

# Run before deployment — every time
protected = ['race', 'gender', 'age', 'CustodyStatus']
audit = shap_bias_audit(shap_values[1], X_test.columns, protected)
# → Rows flagged BIAS RISK → remove from feature set
The Bigger Picture

Explainability is not a nice-to-have. It is the mechanism by which accountability becomes possible. A person denied a loan, flagged as high-risk in a courtroom, or rejected by a hiring algorithm has a legitimate interest in knowing why. SHAP values do not fix bias — removing protected attributes and proxy variables fixes bias. What SHAP does is make the model's reasoning visible, so you can catch problems before deployment, explain decisions to the people affected, and demonstrate to auditors and regulators that the system is operating as intended. The pipeline is not complete until you can explain every decision the model makes.

Accuracy tells you how often the model is right. SHAP tells you why — and that's where the bias hides.

EXPLAINER — 04

WHAT IS EQUALIZED ODDS?

The fairness metric that catches a model treating two groups differently — even when overall accuracy looks fine.

One-Sentence Definition

Equalized Odds requires a model to make equally accurate decisions for every demographic group — meaning its true positive rate and false positive rate must be the same across protected groups like race, gender, or age.

Why This Matters

Most people check model accuracy and call it done. This misses the problem entirely. A model can be 90% accurate overall while still making systematically different types of mistakes for different groups. It might correctly identify qualified candidates in one group at 91%, while only correctly identifying them in another group at 69%. Same model. Same task. Different outcomes — depending on who you are.

This matters because the two types of errors a model makes are not morally equivalent. False negatives — missing someone who should have been flagged — can mean a qualified person is rejected, a sick patient goes undiagnosed, or a creditworthy borrower is denied a loan. False positives — flagging someone who shouldn't have been — can mean an innocent person is detained, or a healthy patient is over-treated. If those errors fall unevenly across demographic groups, the model is discriminating, regardless of what its accuracy score says.

Real-World Proof: COMPAS Analysis

COMPAS doesn't just need to be accurate — it needs to be equally accurate for every defendant, regardless of race. Measuring Equalized Odds reveals what aggregate accuracy hides.

Group True Positive Rate False Positive Rate
Black Defendants 0.89 0.31 ← flagged when innocent
White Defendants 0.67 0.11
Gap 0.22 0.20 — nearly 3× the rate
Black defendants who won't reoffend are falsely flagged as high-risk at nearly three times the rate of white defendants. Aggregate accuracy tells you none of this. Equalized Odds does.
How to Detect It in Python
equalized_odds_audit.py
from fairlearn.metrics import MetricFrame, true_positive_rate, false_positive_rate

metrics = {
  "TPR": true_positive_rate,
  "FPR": false_positive_rate
}

frame = MetricFrame(
  metrics=metrics,
  y_true=y_true,
  y_pred=y_pred,
  sensitive_features=groups # e.g. race, gender
)

print(frame.by_group)
print(frame.difference()) # gap between groups

# Large differences in TPR or FPR = violation
# Run on every protected attribute before deployment

The Math

Two conditions must hold simultaneously. Equal True Positive Rate: P(Ŷ=1 | Y=1, A=a) = P(Ŷ=1 | Y=1, A=b). And Equal False Positive Rate: P(Ŷ=1 | Y=0, A=a) = P(Ŷ=1 | Y=0, A=b). Where Ŷ is the model prediction, Y is the true label, and A is the protected attribute. A model that satisfies one condition but not the other still violates Equalized Odds.

Related Fairness Metrics
Metric What It Requires Trade-off
Equalized Odds Equal TPR + equal FPR across groups May reduce overall accuracy; conflicts with calibration when base rates differ
Equal Opportunity Equal TPR only Relaxed version — lets FPR disparities go unchecked
Demographic Parity Equal positive prediction rates regardless of ground truth Doesn't account for correctness; conflicts with Equalized Odds when base rates differ
Calibration Consistent probability scores across groups Mathematically proven to conflict with Equalized Odds unless base rates are equal
The Bigger Picture

Equalized Odds violations exist because our models are trained on data generated by unequal systems. A criminal justice dataset carries decades of over-policing. A hiring dataset carries decades of gender and racial exclusion. A healthcare dataset carries the structural inequalities of the American insurance system. A model trained on that data learns those inequalities — and unless you actively measure both types of errors across every protected group, the bias remains hidden behind an accuracy score that looks fine.

Equalized Odds is not a cure. Removing protected attributes and proxy variables cures the bias. What Equalized Odds does is make the unequal error rates visible before deployment — so you have something concrete to fix.

Equalized Odds is not a cure. It is a diagnostic. Run it before you deploy.

WHAT IS DISPARATE IMPACT (THE 80% RULE)?

The legal test that has decided employment discrimination cases since 1971 — and the simplest fairness check you can run on any hiring model.

One-Sentence Definition

Disparate Impact is a legal and statistical test that flags a selection process as discriminatory when the selection rate for a protected group is less than 80% of the selection rate for the most-selected group. That threshold is the Four-Fifths Rule, codified in the EEOC's Uniform Guidelines on Employee Selection Procedures (1978), 29 CFR §1607.4(D).

Why This Matters

Most fairness metrics are abstract. The 80% rule is not. It is the actual threshold the EEOC, federal courts, and the Department of Labor use to decide whether to investigate an employer for discriminatory hiring, promotion, or firing. A model can have 95% accuracy and still fail the 80% rule — accuracy says nothing about who is being filtered out at the gate.

This is the rule that matters in practice: HR teams use it to screen vendor hiring tools before deployment. Auditors use it to flag protected-class disparities. Plaintiffs use it as prima facie evidence in Title VII lawsuits. And procurement teams put it in vendor contracts for AI screening products.

How to Calculate It in Python
disparate_impact.py
import pandas as pd

def disparate_impact_ratio(df, prediction_col, group_col, positive_label=1):
  """Four-Fifths Rule: (lowest selection rate) / (highest selection rate)."""
  selection_rates = (
    df.assign(_selected=(df[prediction_col] == positive_label).astype(int))
      .groupby(group_col)["_selected"]
      .mean()
  )
  ratio = selection_rates.min() / selection_rates.max()
  passes = ratio >= 0.80
  return {
    "selection_rates": selection_rates.to_dict(),
    "disparate_impact_ratio": round(ratio, 3),
    "passes_four_fifths_rule": bool(passes),
  }

# Or use Fairlearn — mathematically identical
from fairlearn.metrics import demographic_parity_ratio
ratio = demographic_parity_ratio(y_true, model.predict(X_test),
                                      sensitive_features=groups)
# passes if ratio >= 0.80
The Math

For a protected attribute A taking values a (disadvantaged) and b (advantaged), the selection rate for group g is P(Ŷ=1 | A=g). The Disparate Impact Ratio is DI = P(Ŷ=1 | A=a) / P(Ŷ=1 | A=b), where a is the group with the lower selection rate. The Four-Fifths Rule: DI ≥ 0.80 → no flag; DI < 0.80 → legal scrutiny applies. The ratio always sits in [0, 1] — you always divide the smaller rate by the larger.

Limitations / Trade-offs
Limitation The Problem What to Use Instead
Threshold, not guarantee 0.81 passes and 0.79 fails, but the real disparity is identical Treat it as a floor — investigate any ratio below ~0.90
Ignores error types Measures who gets selected, not who gets selected wrongly Pair with Equalized Odds to catch disparate false positive rates
Satisfiable by lowering the bar Hiring everyone or no one both pass the rule Require job-relatedness analysis alongside the ratio check
Proxy variables defeat it Dropping the protected attribute doesn't fix the ratio if proxies remain Run proxy variable detection first — see Explainer 01
The Bigger Picture

The 80% rule is one of the few places where civil-rights law, statistics, and machine learning all agree on the same number. It was written in 1978 for paper-and-pen hiring tests; it applies word-for-word to a Random Forest classifier in 2026. That is what makes it useful as a deployment check: it is not a research metric. It is the metric that triggers an EEOC investigation, that survives the motion-to-dismiss stage of a Title VII lawsuit, and that procurement teams put in vendor contracts.

If your hiring model fails it, you do not need a fairness expert to explain why that is a problem — the law has already explained it for you.

Run the 80% rule against every classifier that decides who gets a job, a loan, a lease, or an insurance policy. If it fails, you have a legal problem before you have a technical one.

EXPLAINER — 06

WHY FAIRNESS METRICS CONFLICT

You can't be fair to everyone at once — and the math proves it. Any choice of fairness metric is also a choice about who bears the cost of being wrong.

One-Sentence Definition

Fairness metric conflict is the mathematically proven impossibility of satisfying multiple fairness criteria simultaneously when base rates differ across demographic groups — meaning any choice of fairness metric is also a choice about who bears the cost of being wrong.

Why This Matters

Most people assume that making an AI "fair" is a single, solvable problem. It isn't. Researchers have formally proven — not just observed, but proven — that several common fairness definitions are mutually incompatible except in degenerate edge cases.

Courts, regulators, and hiring committees each invoke different fairness definitions — and they are often in direct mathematical conflict. A model optimised to satisfy one metric will necessarily perform worse on another. Choosing a metric is an ethical and political decision, not a technical one. If you deploy a model and declare it "fair" without specifying which metric you used and why, you have not solved the fairness problem. You have hidden it.

The Three Metrics That Can't All Be True
Metric What It Requires Used In
Demographic Parity P(Ŷ=1 | Group=A) = P(Ŷ=1 | Group=B) — equal positive outcome rates Hiring, lending, admissions audits. Underpins the 80% rule.
Equalized Odds Equal TPR and equal FPR across groups — equal error rates in both directions Criminal risk assessment, medical screening. ProPublica's COMPAS critique.
Predictive Parity P(Y=1 | Ŷ=1, Group=A) = P(Y=1 | Ŷ=1, Group=B) — equal accuracy of flags Credit scoring, recidivism prediction. Northpointe's defence of COMPAS.
Chouldechova (2017) and Kleinberg et al. (2016) independently proved: when base rates differ between groups, you cannot simultaneously satisfy equalized odds and predictive parity — except when the model is perfect or predicts at random.
Real-World Proof: The COMPAS Dual-Reading

ProPublica and Northpointe were both right about COMPAS — using different metrics. This is the clearest public illustration of the impossibility.

STEP 02
The full audit function — compute all three metrics simultaneously
full_fairness_audit.py
from sklearn.metrics import confusion_matrix

def full_fairness_audit(y_true, y_pred, groups):
  """TPR, FPR, and PPV per group — run before deployment."""
  results = {}
  for group in groups.unique():
    mask = groups == group
    tn, fp, fn, tp = confusion_matrix(y_true[mask], y_pred[mask]).ravel()
    results[group] = {
      'TPR': round(tp / (tp + fn), 3) if (tp + fn) > 0 else None,
      'FPR': round(fp / (fp + tn), 3) if (fp + tn) > 0 else None,
      'PPV': round(tp / (tp + fp), 3) if (tp + fp) > 0 else None,
      'Pos Rate': round((tp + fp) / (tp + fp + tn + fn), 3),
    }
  return pd.DataFrame(results).T

# Run on every model before deployment
audit = full_fairness_audit(y_true, y_pred, groups=test_df['race'])
# If any metric diverges significantly across groups → investigate

Metric What It Prioritises Who Bears the Cost When It Fails
Demographic Parity Equal access to positive outcomes Groups with higher true positive rates may be under-predicted
Equalized Odds Equal error rates in both directions Predictive accuracy per group may be sacrificed
Predictive Parity Equal reliability of positive flags Groups with lower base rates face higher false positive rates
Step 1 — Audit Base Rates Before Picking a Metric
base_rate_audit.py
def base_rate_audit(df, outcome_col, group_col):
  """If base rates differ, metric conflicts are guaranteed."""
  rates = df.groupby(group_col)[outcome_col].mean().round(4)
  ratio = rates.max() / rates.min()
  if ratio > 1.2:
    print(f"⚠ Base rates differ by {ratio:.2f}x — metric conflicts will arise.")
  return rates

The Bigger Picture

The COMPAS dataset shows Black defendants reoffending at higher rates. But that statistic is built on arrest records — and arrest rates reflect policing patterns, not actual crime rates. A neighbourhood that is over-policed produces more arrests of its residents, inflating their apparent recidivism rates. The model treats this as ground truth and calibrates to it. Predictive parity holds — but against a corrupted baseline.

This is why choosing a metric is not sufficient. You must also audit the data that defines your ground truth. There is no universally correct metric. The choice depends on the domain, the asymmetry between error types, and a normative judgement about whose interests to protect. Document your choice in the model card. A model without a declared fairness metric has not been audited — it has been deployed.

Both ProPublica and Northpointe were right. They were measuring different things. Pick your metric deliberately, document the trade-off, and never call a model "fair" without saying fair by whose standard.
